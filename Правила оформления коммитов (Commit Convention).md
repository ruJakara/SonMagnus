# Правила оформления коммитов (Commit Convention)

Мы используем стандарт **Conventional Commits** для структурирования истории изменений. Это позволяет легко генерировать changelog и понимать, что изменилось в проекте.

Каждое сообщение коммита должно иметь следующий формат:

```
<тип>(<область>): <краткое описание>

[Тело сообщения]

[Нижний колонтитул (например, BREAKING CHANGE, Closes #123)]
```

## 1. Тип (`<тип>`)

Тип — это обязательное слово, которое описывает характер изменения.

| Тип | Описание |
| :--- | :--- |
| `feat` | **Новая функция.** Введение новой возможности или фичи. |
| `fix` | **Исправление ошибки.** Устранение бага. |
| `docs` | Изменения в документации (README, COMMIT_CONVENTION и т.д.). |
| `style` | Изменения, не влияющие на смысл кода (форматирование, пропущенные точки с запятой, пробелы). |
| `refactor` | Изменение кода, которое не исправляет ошибку и не добавляет функцию (переименование, реструктуризация). |
| `perf` | Изменение кода, улучшающее производительность. |
| `test` | Добавление или исправление тестов. |
| `build` | Изменения, влияющие на систему сборки или внешние зависимости (npm, Godot export settings). |
| `ci` | Изменения в файлах и скриптах CI (Continuous Integration). |
| `chore` | Прочие изменения, не связанные с кодом приложения (обновление .gitignore, рутинные задачи). |

## 2. Область (`(<область>)`)

Область — это необязательное слово, указывающее, какая часть проекта была затронута. Используйте названия сцен, систем или компонентов.

**Примеры:** `(combat)`, `(ui)`, `(player)`, `(inventory)`, `(main_menu)`.

## 3. Краткое описание (`<краткое описание>`)

Краткое описание — это обязательное, краткое изложение изменения.

*   Используйте повелительное наклонение (например, "Добавить", "Исправить", "Обновить"), а не прошедшее ("Добавил", "Исправил").
*   Начинайте с заглавной буквы.
*   **Не ставьте точку в конце.**
*   Максимальная длина — 50 символов.

## 4. Тело сообщения

Тело сообщения — это необязательная, но рекомендуемая часть для более подробного описания изменения.

*   Используйте повелительное наклонение.
*   Опишите мотивацию для изменения и сравните с предыдущим поведением.

## 5. Примеры

### Новая функция
```
feat(combat): Добавить систему блокирования урона

Введена новая механика блокирования урона щитом.
Успешная блокировка снижает входящий урон на 50% и
расходует 10 единиц выносливости.
```

### Исправление
```
fix(ui): Исправить утечку памяти при закрытии инвентаря

Объект инвентаря теперь корректно удаляется из памяти
после вызова `queue_free()`.
```

### Критическое изменение (Breaking Change)
Если коммит вносит критическое изменение, которое может сломать работу других частей проекта, его необходимо явно указать в нижнем колонтитуле.

```
refactor(player): Изменить API метода move_and_slide

BREAKING CHANGE: Метод `Player.move_and_slide()` теперь принимает
только один аргумент `velocity` вместо трех (`velocity`, `direction`, `delta`).
Необходимо обновить все вызовы этого метода.
```
